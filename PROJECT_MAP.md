## 1. Project Overview

**What**: A real-time audio effect plugin (guitar distortion / amp simulation) targeting the **VST3**, **CLAP**, and **standalone** formats.

**Purpose**: Process live audio through a chain of linear DSP (EQ, convolution) and neural saturation (Mojo-powered tanh non-linearity) with zero additional latency and zero allocations on the audio thread.

### 2.2 Dual-Target Architecture

The codebase produces **two binaries** from a single crate:

| Target | Crate Type | Entry Point | UI Backend | Audio I/O |
|---|---|---|---|---|
| **Plugin** (VST3/CLAP) | `cdylib` | `src/lib.rs` → `nih_export_clap!()` / `nih_export_vst3!()` | nih_plug_egui (DAW-embedded) | DAW host |
| **Standalone** | `bin` | `src/bin/standalone.rs` → `main()` | eframe/egui (native window) | CPAL (ALSA/JACK/etc.) |

Both share `src/core/` modules (`dsp`, `state`, `ui`) but implement **independent DSP pipelines**:

- **Plugin mode**: `BaseIO::process()` in `lib.rs` — uses nih_plug's `Buffer` abstraction, parameter smoothers, `ProcessContext`.
- **Standalone mode**: `audio_worker()` thread in `standalone.rs` — raw CPAL input/output streams, manual interleaving, `RingBuffer`-based pass-through.

### 2.3 FFI Bridge Pattern

The `src/bridge/` module defines a uniform `ExternalProcessor` trait that both backends implement:

```rust
// src/bridge/mod.rs (13 lines)
pub trait ExternalProcessor {
    fn init(&mut self, sample_rate: f32);
    fn process_block(&mut self, buffer: *mut f32, length: usize);
}
```

**Faust bridge** (`src/bridge/faust.rs`, 69 lines): Uses `bindgen`-generated bindings from `dsp/wrapper.h`. The opaque `FaustHandle` wraps a C++ `FaustInstance` (contains `mydsp*` + `ParamMapUI*`). Parameters are set via individual C FFI functions (`faust_set_eq_low_freq`, etc.) that use a fast string-keyed map lookup.

**Mojo bridge** (`src/bridge/mojo.rs`, 65 lines): Declares `extern "C"` functions `mojo_init()` and `mojo_process_block()`. The critical zero-copy path converts the `*mut f32` buffer pointer to `usize` before crossing the FFI boundary — the Mojo side reconstructs the pointer from the integer address.

## 4. Processing Pipeline

The audio pipeline runs **single-pass, in-place, per-channel** in `BaseIO::process()` (`src/lib.rs`, lines ~230-340). All processing is synchronous and zero-latency (`set_latency_samples(0)`).

### Stage 0: Input Routing

**File**: `src/lib.rs:233-258`  
**What it does**: Based on `InputSelect` enum (Stereo / Input1-Mic / Input2-Guitar), the raw stereo input is collapsed or preserved:

| Mode | Left Channel | Right Channel |
|---|---|---|
| `Stereo` | `buffer[0]` (unchanged) | `buffer[1]` (unchanged) |
| `Input1` | `buffer[0]` | `buffer[0]` (copy of left) |
| `Input2` | `buffer[1]` | `buffer[1]` (copy of right) |

This is applied **before any DSP** by iterating `buffer.iter_samples()`, reading both channels, and writing the selected values back. The result is an in-place transformation of the input buffer.

### Stage 1: Faust 3-Band Parametric EQ

**Rust**: `src/lib.rs:268-285` (EQ params set + `faust.process_block()`)  
**Faust DSP**: `dsp/main.dsp:4-26`  
**C Wrapper**: `dsp/wrapper.cpp:47-130`  
**Filters**: `faust-ddsp/filters.lib` (SVF-based `low_shelf`, `peak_eq`, `high_shelf`)

The EQ has 3 bands, each with frequency, gain (±12 dB), and Q controls:

| Band | Filter Type | Default Freq | Range | Q Range |
|---|---|---|---|---|
| Low | `low_shelf` | 100 Hz | 20–1000 Hz | 0.707–10.0 |
| Mid | `peak_eq` | 1000 Hz | 100–10,000 Hz | 0.707–10.0 |
| High | `high_shelf` | 5000 Hz | 1000–20,000 Hz | 0.707–10.0 |

All filters use **State Variable Filter (SVF)** topology from `filters.lib` for stability during real-time parameter modulation. The SVF uses trapezoidal integration to prevent zipper noise.

The EQ chain is: `eq_low : eq_mid : eq_high : ma.tanh` — a series cascade followed by soft clipping.

**Conditional**: Skipped if `eq_active` is false.

### Stage 2: Pre-EQ Convolution (LTI Filter)

**File**: `src/lib.rs:288-294`  
**Asset**: `neural/drive/pre_eq_ir.wav`  
**Crate**: `fft-convolver` (v0.3.0)

A linear time-invariant filter applied via FFT convolution. The impulse response is loaded from a WAV file during `initialize()`. The convolution is performed using `FFTConvolver::process()` which handles partition-based FFT convolution internally.

The processing flow:
1. Copy current buffer to `temp_buffer` (backup)
2. Call `pre_eq_convolver.process(temp_slice, output_slice)`
3. On error, restore from `temp_buffer`

**One convolver instance per channel** (L/R).

### Stage 3: Mojo Neural Drive (Saturation)

**Rust**: `src/lib.rs:297-303`  
**Mojo**: `neural/main.mojo:8-35`  
**Bridge**: `src/bridge/mojo.rs:50-62`

The neural non-linearity — a polynomial approximation of `tanh()`:

```
tanh(x) ≈ x * (27 + x²) / (27 + 9 * x²)
```

Implementation:
1. Input clamped to [-4.0, 4.0] to avoid overflow
2. Polynomial applied sample-by-sample
3. Input scaled by `drive` (pre-gain, 0–30 dB range)
4. Output scaled by `output_gain` (makeup gain, -24 to +12 dB)

**Zero-copy**: The buffer pointer is cast to `usize` in Rust, passed to Mojo, and reconstructed as `UnsafePointer[Float32, MutAnyOrigin]`. No memory allocation on the audio thread.

**Conditional**: Skipped if `neural_amp_active` is false.

### Stage 4: Cabinet IR Convolution

**File**: `src/lib.rs:306-312`  
**Asset**: `neural/drive/cabinet_ir.wav`  
**Crate**: `fft-convolver`

Same mechanism as Stage 2 — partition-based FFT convolution with a speaker cabinet impulse response. The WAV is loaded during `initialize()` and convolved per-channel.

### Stage 5: Master Gain

**File**: `src/lib.rs:315-318`  
**Parameter**: `gain` (FloatParam, -30 to +30 dB, logarithmic smoothing)

Simple per-sample multiplication of the entire buffer by the smoothed gain value.

### Stage 6: Neural Volume

**File**: `src/lib.rs:321-325`  
**Parameter**: `neural_amp_volume` (FloatParam, -24 to +12 dB, logarithmic smoothing)

Additional post-processing gain applied only when neural amp is active. Scaled by the smoothed volume parameter.

### Stage 7: NaN/Inf Sanitization (Always Active)

**File**: `src/lib.rs:329-334`

Every sample is checked for `NaN` or `Infinity` and replaced with `0.0`. This runs **even in bypass mode** to prevent audio hardware damage from invalid samples.

### Analyzer Feed

**File**: `src/lib.rs:337-343`

After all processing, the final audio samples are mixed (L+R)/2 and pushed into a `RingBuffer` (64K capacity) for the spectrum analyzer visualization thread.

### Parameter Smoothing

All `FloatParam` instances use `SmoothingStyle::Logarithmic(50.0)` or `Linear(50.0)` (50 ms time constant). The **smoothed values** are read via `.smoothed.next()` in the audio thread — never `.value()` directly. This prevents parameter zipper noise.

## 6. Key Design Patterns

### 6.1 Zero-Copy FFI (Rust ↔ Mojo)

**Problem**: Mojo 0.26+ does not allow `UnsafePointer` as a parameter to `@export` functions due to parametricity restrictions.

**Solution — Address Bypass Pattern**: Pass the memory address as an integer and reconstruct the pointer on the receiving side.

**Rust side** (`src/bridge/mojo.rs:55-59`):
```rust
let ptr = buffer as usize;          // Convert *mut f32 → usize
mojo_process_block(ptr, length, self.drive, self.output_gain);
```

**Mojo side** (`neural/main.mojo:16-17`):
```mojo
var data = UnsafePointer[Float32, MutAnyOrigin](unsafe_from_address=address)
```

This is the **only pattern that works** for Mojo FFI in shared libraries. Do not use `@extern_c` — use `@export`. The buffer is modified in-place; no data is copied across the FFI boundary.

### 6.2 ExternalProcessor Trait

**File**: `src/bridge/mod.rs`

A uniform interface that abstracts all external audio processors:

```rust
pub trait ExternalProcessor {
    fn init(&mut self, sample_rate: f32);
    fn process_block(&mut self, buffer: *mut f32, length: usize);
}
```

Both `FaustProcessor` and `MojoProcessor` implement this trait, allowing the main pipeline to treat them interchangeably. The trait enforces the in-place, non-allocating contract required for real-time audio.

### 6.3 Faust C Wrapper Pattern

**Files**: `dsp/wrapper.h` + `dsp/wrapper.cpp`

The Faust DSP engine is wrapped in a C ABI via an opaque pointer pattern:

1. **Opaque handle**: `typedef void* FaustHandle` hides the C++ internals from Rust.
2. **Lifecycle**: `faust_create()` → `faust_init(rate)` → `faust_process(buf, len)` → `faust_destroy()`
3. **Parameter map**: `ParamMapUI` implements Faust's `UI` interface, capturing `float*` pointers into a `std::map<std::string, float*>`. Parameter setters (`faust_set_eq_low_freq`, etc.) do a single map lookup + pointer write — O(log n) but constant for 9 params.
4. **Dual-mono**: Faust's stereo `process` signature is `(_,_) : (eq_chain, eq_chain)`. The wrapper passes the same buffer as both input and output channels, effectively running the same processing on the mono buffer.

### 6.4 Parameter Smoothing

**File**: `src/core/state/plugin_params.rs`

Every `FloatParam` in `BaseIOParams` is configured with a 50 ms smoother:

```rust
FloatParam::new(...)
    .with_smoother(SmoothingStyle::Logarithmic(50.0))  // for dB/Hz params
    // or
    .with_smoother(SmoothingStyle::Linear(50.0))        // for Q/gain params
```

Smoothing styles per parameter:

| Parameter | Smoothing | Reason |
|---|---|---|
| `gain`, `neural_amp_volume`, `neural_drive`, `neural_output_gain` | Logarithmic(50ms) | dB-scaled values |
| `eq_low_freq`, `eq_mid_freq`, `eq_high_freq` | Logarithmic(50ms) | Frequency values |
| `eq_*_gain` (all) | Linear(50ms) | Linear dB range |
| `eq_*_q` (all) | Linear(50ms) | Q factor |

In the audio thread, **always** call `.smoothed.next()` — never read `.value()` directly, as that would bypass the smoothing envelope and cause audible artifacts (clicks/pops).

### 6.5 RingBuffer-Based Analyzer Feed

**File**: `src/lib.rs:337-343` + `src/core/dsp/analyzer.rs:39-89`

The audio thread pushes (L+R)/2 mixed samples into an `rtrb::RingBuffer` (lock-free, single-producer-single-consumer). The UI thread drains the consumer in `editor()`, feeding the `AnalyzerDsp`.

The analyzer performs:
1. Collect 2048 samples → `in_buffer`
2. Apply **Blackman-Harris window** (4-term): `0.35875 - 0.48829*cos(x) + 0.14128*cos(2x) - 0.01168*cos(3x)`
3. Forward FFT (real-to-complex via `realfft`)
4. Magnitude → dB conversion: `20*log10(mag / (FFT_SIZE/2))`
5. **Pink noise tilt correction**: `db += 3*log2(freq/1000)` compensates for natural spectral rolloff
6. **Exponential moving average**: `spectrum[i] = prev*0.85 + new_db*0.15` for smooth animation

### 6.6 Plugin ↔ Standalone Shared UI

**File**: `src/core/ui/main_view.rs`

The `render_shared_panels()` function is used by both the plugin (`lib.rs`) and standalone (`standalone.rs`). It accepts closures for control rendering, making it UI-framework-agnostic within the project. The same spectrum visualizer, signal chain, and knob layouts appear in both modes.

### 9.4 Architecture Decisions Record

| Decision | Rationale | Date implied |
|---|---|---|
| Mojo replaces tract-onnx for inference | Synchronous zero-copy ≈ zero latency; no background threads needed | 2026-03 |
| Faust SVF over biquad filters | State Variable Filters are stable under real-time modulation; no coefficient recalculation artifacts | 2026-03 |
| Dual-mono processing (not true stereo) | Simpler implementation; each channel processed independently through identical chains | Initial |
| FFTConvolver over direct convolution | O(n log n) for long impulse responses; partition-based for low latency | Initial |
| egui over vizia for standalone | Simpler API; same UI code shared with nih_plug_egui plugin mode | Initial |
| Address Bypass Pattern for Mojo FFI | Required by Mojo 0.26+ `@export` parametricity constraints | 2026 |
